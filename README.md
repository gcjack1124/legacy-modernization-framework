# Legacy Modernization Framework

> A multi-agent AI workflow for reverse-engineering and rewriting legacy enterprise
> systems — with a built-in, self-auditing quality gate.
>
> 一套以**多 Agent AI 工序鏈**為核心的遺留系統現代化方法論：把 4GL、VBA 等老舊企業程式，
> 逆向工程並改寫成現代 SQL / Python，過程內建 **15+1 維度自證審查**品質閘門。

> 方法論三份文件（9 階段工序鏈 / 15+1 維度 audit / 本 README）已完成；去敏化案例卷宗持續補上。

## The Problem | 問題

很多公司的 IT 裡都躺著一批年代久遠的 legacy 4GL / VBA 老程式：沒有文件、邏輯散在好幾千行裡。沒人敢動，也沒人能改寫。這套框架想解的，就是**怎麼用 AI 把這種系統安全、又能驗證地現代化**。

難的地方其實不是「把語法翻譯過去」，而是**在沒人完全看懂原邏輯的情況下，還要證明改寫後的結果跟舊系統一模一樣**。只要漏掉一個條件、差一個邊界，程式照樣會跑、報表照樣印得出來，錯誤往往要等到月結對不上帳才爆出來——到那時候，要追是哪一行出問題已經很難了。

傳統的兩條路，各有各的死角：

- **人工一行一行改**——慢，而且人也會看漏，沒有一套機制幫你防呆。
- **整包丟給 AI 一次生成**——快，但「看起來對」沒辦法驗證；而且同一個 AI 又寫又審，等於自己改自己的考卷，有天生的盲點。

這套框架賭的是第三條路：**用多個 agent 分工，加上強制的驗證閘門**，讓「AI 的速度」和「結果可驗證」同時成立，而不是二選一。

## Proof | 實證數據

| 指標 | 數據 |
|------|------|
| 改寫規模 | axcp500：4,594 行 4GL → 5,410 行 SQL Server SP（6 支拆分，含寫入 8 張表的月結成本計算邏輯）|
| 學習曲線 | 同一支報表兩次改寫：第一次 17 輪 / 2 天 → 第二次 0 輪 / 3 小時（脈絡見下）|
| 已交付驗收 | apmr554：4GL 報表組改寫上線、100% 對齊關鍵筆、產銷主管實際使用 |
| 品質閘門 | 15+1 維度自證 audit |

**數據脈絡（誠實版）：**

- 樣本還小（兩個真實案例），上面的數字反映的是「方法論慢慢成形 + 教訓累積」的學習曲線，**不是統計基準**。
- **17 輪 → 0 輪（同一支報表、改了兩次）**：axcp500 第一次（離線版）是在工序鏈還沒成形時做的，前後修了 17 輪、花了 2 天；第二次（連線版）套上成形的流程、又有第一次的教訓墊底，3 小時就零返工跑完。**不過要老實講**：第二次也因為砍掉了一部分範圍（不寫回 Informix、跳過約 625 行回填邏輯）而比較輕。所以「16×」是方法論變熟、教訓複用、範圍縮小三件事加起來的，不是單純流程變強。
- **驗證到哪（老實說）**：axcp500「改寫 + 部署 + 全量跑通」都做到了，但**跟舊系統的 1:1 對齊驗證還沒跑完**（卡在 ERP 環境和時程）。所以它證明的是「能逆向這種複雜度」跟「學習曲線」，**沒有宣稱結果已經驗證正確**。真正完整對齊、被主管收下的是 apmr554（見 [Case Studies](#case-studies--案例)）。
- **目前的侷限**：實證集中在「4GL → SQL Server SP」這一塊；VBA 之類的其他來源是設計上能延伸，但**還沒驗證過**。

## How to Use | 如何使用

這是一份**方法論參考**，不是裝了就能跑的工具——repo 裡放的是流程和檢核框架的文件。要套到自己的 legacy 改寫，大致是三步：

1. **解析來源** — 先把 legacy 程式拆成結構化的交付物（輸出欄位、篩選條件、邏輯單元對照），之後每一關都以它為唯一依據。
2. **跑 9 階段工序鏈** — 設計／改寫／審查／驗收各由獨立角色負責，上一關交出的東西鎖成唯讀，出錯有明確的回溯規則。見 [`docs/9-phase-workflow.md`](docs/9-phase-workflow.md)。
3. **過 15+1 維度 audit** — 當品質閘門用，每個維度要交出自證證據、不能只打勾，最後再加一層 meta 自審防止流於形式。見 [`docs/sp-audit-framework.md`](docs/sp-audit-framework.md)。

> 這套參考實作是跑在 Claude Code 的多 agent 環境上，但原則本身跟工具無關——只要環境能分出獨立的 agent 角色就適用。
> **人工簽核閘門是刻意留著的**：A 階段沒簽核就擋下，F 階段一定要人親自跑過才算數。AI 負責加速，但不負責把人的把關拿掉。

## Case Studies | 案例

這套方法論靠兩個互補的真實案例撐著——一個是「已經交付、驗證過、被收下」，一個是「最複雜、用來看學習曲線」：

- **apmr554（已交付 + 主管驗收）** — 一組 legacy 4GL 報表（採購／交貨／付款天數）逆向後改寫成 SQL Server SP + Python，端到端把報表產出來、**關鍵筆 100% 對齊**、上線給產銷主管實際使用，回饋是「程式很好用」。這是「**真的算對、也真的有人在用**」的證據。（這個案子做在完整 9 階段工序鏈成形之前，算是方法論還沒定型時的實戰。）
- **axcp500（最複雜 + 學習曲線）** — 一支大約 4,594 行的 legacy 4GL 月結成本計算引擎（有寫入邏輯、有副作用，不是純讀的報表）。已經改寫成 6 支 SP、部署、全量跑通；但**跟舊系統的 1:1 對齊驗證還沒完成**（卡 ERP 環境）。它的價值在於「能逆向這種複雜度」跟「**17 輪 → 0 輪的學習曲線**」。完整側寫見 [`docs/case-axcp500-walkthrough.md`](docs/case-axcp500-walkthrough.md)。
