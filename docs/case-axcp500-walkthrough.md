# Case Walkthrough — axcp500：4GL 月結成本計算引擎 → SQL Server SP

> 一支約 4,594 行的 legacy 4GL **月結成本計算引擎**，用 9 階段工序鏈 + 15+1 維度 audit 改寫成 SQL Server Stored Procedure 的實戰側寫。
> 重點不在「改寫了多少行」，而在 **audit 擋下了哪些「單一 AI 會安靜放過」的錯**。

## 背景 | Context

axcp500 不是純讀報表，而是含**寫入邏輯與副作用**的複雜程式——會寫多張表、跑實際成本制計算、依工單入庫狀態做翻轉。原始開發者已離職、缺乏文件、邏輯散落在數千行程序裡。

它被選為案例，正因為它夠難：是「AI 到底能不能逆向**有副作用的複雜企業程式**」的試金石。目標不是「跑得出數字」，而是改寫成 SQL Server SP **且能證明結果與舊系統一致**。

## 過程 | How It Ran

照 [9 階段工序鏈](9-phase-workflow.md)走：4GL 解析成結構化交付物（輸出欄位、篩選條件、邏輯單元對照）→ Architect 建「原始 ↔ 改寫」對照表、PM 訂驗收標準 → Coder 依鎖定設計改寫成 6 支 SP → Reviewer 跑驗證規則 + 15+1 維度 audit → 人工實跑。

真正的價值不在 happy path，而在 **audit 在進入下一關前擋下的東西**。以下三個都是「單一 AI 自己審自己時會放過、但跨角色 + 多維度檢核擋下」的真實案例。

## audit 擋下的三個「會安靜出錯」的坑 | Three Silent Failures the Audit Caught

### ① NULL 連環修 —— 維度 3（NULL handling）

改寫後的 SP 在寫入目標表時，沒有逐一補齊所有 `NOT NULL` 欄位。這類錯的陰險之處是：**修好一處、另一處又冒出來**，演變成「修 A 壞 B、修 B 壞 C」的連環 patch，輪數失控。

單一 AI 一次只盯得到眼前那一處；**維度 3 強制「逐一對照目標表每一個 NOT NULL 欄位」**，把整批一次揪出，而不是一處一處撞。

### ② 定長字元的尾端空白 —— 維度 12（char(N) 比對防護）

來源端的定長字元欄位（`char(N)`）會帶**尾端空白**。比對時若沒有兩邊都 trim，篩選會 **silent fail——回 0 筆、而且不報錯**。這是最難抓的一種：程式跑得很順、就是結果默默全空。

**維度 12 專門擋這個**：任何定長字元欄位用於 `=` / `LIKE` / `IN` / `JOIN ON`，兩邊都必須 `LTRIM(RTRIM())`。

### ③ 業務參數的「曆制」沒宣告 —— 維度 14 的誕生

一個年份入參，用的是哪種曆制（民國 vs 西元）**沒有明文宣告**。下游的 validation range 因此套錯制，期間檢查變成**恆為真或恆擋**——又是一個「跑得動、結果默默錯」。

最關鍵的是：這個錯被抓到時，**當時所有 audit 維度全聚焦在「SQL 內部正確性」，竟沒有一條涵蓋「業務參數的語意層」**。於是框架因此**長出維度 14（業務參數語意對齊）**：每個入參的編碼／單位／制度必須明文宣告，並用來源端的實際值佐證、不能只信 schema 文件。

> 這一條是整套框架「**被真實的坑教會、自己長出新維度**」的最好例子——它不是憑空設計出來的 checklist，是撞出來的。

## 學習曲線 | The Learning Curve

同一支 axcp500 改寫了**兩次**：

- **第一次（離線版）** — 方法論還沒成形，靠反覆撞錯累積教訓：**17 輪修補、2 天**。
- **第二次（連線版）** — 套用成形後的工序鏈 + 前案教訓：**3 小時、零返工**。

> 誠實註：第二次快，**不純是方法論變強**。連線版也砍掉了部分範圍（不寫回 Informix、跳過約 625 行回填邏輯）。「16×」是**方法論成熟 + 教訓複用 + 範圍縮小**三者疊加，不是單一因素。

## 結果 | Outcome（誠實版）

- **已完成**：4GL 解析 → 6 支 SP 設計、改寫、部署到 SQL Server 2008 R2；全量跑通一次完整月結資料。
- **尚未完成**：與舊系統的 **1:1 對齊驗證（筆數一致 + 累計差 < 1 元）尚未跑完**，卡在 ERP 環境/時程；另有效能（單次全量 11+ 小時）待優化。
- 依框架自己的原則，未驗證的部分標 `UNTESTED`——**合法狀態，但不算 done**。

所以本案的定位是 **「複雜度 + 學習曲線」的證據，不宣稱正確性已驗**。要找「已完整對齊 + 主管驗收」的證據，見 [apmr554](../README.md#case-studies--案例)。

> 把「還沒驗完」誠實標出來，本身就是這套框架要傳達的態度：**`UNTESTED` 不可以偷偷當成 `PASS`。**
