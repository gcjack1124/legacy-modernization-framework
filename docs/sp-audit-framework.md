# 15+1 Dimension Self-Audit Framework | 十五加一維度自證審查框架

> 讓 AI 自我審查程式改寫品質的領域檢核框架——15 個技術維度 + 1 層 meta 自審。
> A domain checklist that forces an AI to audit its own rewrite across 15 technical dimensions, plus one meta layer that audits the audit itself.

## Why Self-Audit | 為什麼需要自證審查

AI 改出來的程式「看起來對」，不代表「真的對」。比較麻煩的還不是它寫錯，而是它往往**一次只暴露一個問題**：先發現某欄 NULL 沒處理、補好；接著才發現型別對不上、再補；然後又冒出字元比對的問題。而且修好一處常常牽動另一處——修 A 壞 B、修 B 壞 C，輪數一直疊上去。

所以這套框架的做法有三層：

1. **一次就把該查的多個維度全攤開來看**，而不是想到什麼查什麼——把「下次才會想到」的面向，這次就強制納進來。
2. **每個維度都要交出自證證據**：不是打一個 ✅，而是寫清楚「我用什麼方法查的、查了什麼、發現幾條」。打勾很好造假，要寫出方法就難一些。
3. **再加一層 meta 自審**，專門抓「audit 本身流於形式」——避免檢核退化成「看到維度標題就套標準答案」。

## The 15 Dimensions | 十五個維度

每個維度背後都是一類真實踩過的坑。下面是去敏化後的通則（具體表名／欄位／行號都拿掉了）。

| # | 維度 Dimension | 檢核重點 |
|---|----------------|----------|
| 1 | 語法／版本相容性 | 目標 DB 與來源 DB 各自支援的語法／函式不混用，依實際版本 baseline 檢查 |
| 2 | Schema 對齊 | 引用欄位真實存在、型別與用法對齊、跨階段／跨模組的暫存表接得上 |
| 3 | NULL handling | NOT NULL 欄位 INSERT 補齊（含 MERGE）、可能為 NULL 的來源（SUM／JOIN／變數）包 `ISNULL`、逐一對照目標表 NOT NULL 欄位 |
| 4 | 暫存表／變數寬度 + scope | 寬度 ≥ 來源欄位、跨 SP scope 正確（誰建誰讀）、同名衝突防護 |
| 5 | 控制流結構 | `IF...BEGIN` 區塊不被夾斷、`AND` 不假設 short-circuit、游標 LOCAL+釋放、變數先宣告後用 |
| 6 | dynamic SQL 結構 | 超長 `UNION ALL` 拆段（避免 parser 行號偏移）、參數化不靠字串拼接、字串內留除錯標記 |
| 7 | 錯誤處理 | `CATCH` 不吞掉原始撞點、`RAISERROR` 後確實終止、`TRY` 邊界不含成功路徑 |
| 8 | debug-friendly | 每段留 `PRINT` marker、型別對齊註解、來源行號 line comment——讓出錯時能快速 narrow |
| 9 | 跨庫橋接：字串 escape〔連線版〕 | 多層引號嵌套正確、動態拼接做 escape、防跨層 SQL injection |
| 10 | 跨庫橋接：來源 DB 方言〔連線版〕 | 橋接查詢內只用「來源 DB」支援的函式／語法，不套用目標 DB 的函式，也不假設來源是較新版方言 |
| 11 | 跨庫橋接：transaction 限制〔連線版〕 | 不以分散式交易包住橋接寫入、橋接內純讀、寫入只包目標 DB 的表 |
| 12 | 定長字元比對防護 | `char(N)` 來源用於比對（`=`／`LIKE`／`IN`／`JOIN ON`）兩邊都 trim，否則 trailing space 會 silent fail 回 0 筆 |
| 13 | 暫存表生命週期 + 命名衝突 | 預設先 DROP 再 CREATE；跨 SP 共用才用 `IF OBJECT_ID IS NULL` 守護；同一 SP 內不重名 |
| 14 | 業務參數語意對齊 | 每個入參的**編碼／單位／制度**必須明文宣告（如年份用哪種曆制、金額幣別、計量單位），並有來源端**實際值**佐證（不只信 schema 文件）；validation range 與宣告一致；多處聲明交叉一致 |
| 15 | SP 反模式（SP smells） | 借 ISTQB／xUnit「test smells」概念的 SP 版：回傳欄過多無法定位、偷依賴外部物件、硬編碼／不可重現、跨 SP 隱性順序依賴 |

> 〔連線版〕＝跨庫橋接（linked-server）架構才適用的維度；單庫任務可標「⏭️ 跳過」。

## +1 — Meta：殺蟲劑悖論自審 | The Pesticide-Paradox Self-Check

> 借自 ISTQB 七大測試原則之一：「同一組測試重複跑會逐漸失效」。
> 這一層的目的不是抓程式的 bug，而是抓 **audit 本身流於形式**。

跑完 15 維度後，**強制回答 3 問**（不可省略、不可答「無」）：

1. **形式化檢查** — 這次 audit 有沒有「看到維度標題就套標準答案、沒實質檢查」的維度？是哪幾個？
2. **盲區探索** — 列出 3 件「15 維度沒涵蓋、但這支程式直覺上該擔心」的事。
3. **未來推估** — 如果這支程式上線一個月內出 bug，憑直覺猜會是哪一段？為什麼直覺指向那？

任一答是「沒有／無／不知道」→ 紅旗，標註「audit 可能流於形式，建議換維度組合或找新角度」。

## 兩個讓它不只是「打勾」的機制 | What Makes It More Than a Checkbox

1. **自證涵蓋表（self-evidence table）** — 每個維度結尾都要附一行：`涵蓋 ✅/⏭️ | 用什麼方法查 | 發現幾條`。逼出實質檢查——打勾很好造假，要寫出「用什麼方法查的」就難一些（**但也不是絕對：AI 一樣會編漂亮的假證據，所以真正的底線是靠機器、不是靠 AI 的良心——見下面「機器查核才是底線」和「已知限制」**）。
2. **不可後補** — meta 自審一定要在 15 維度跑完的當下立刻寫。拖越久，人越會替自己找理由，自審就失去意義了。

## 不只是「AI 自己說有查」｜機器查核才是底線 | Machine Checks, Not AI Honesty

最該被質疑的問題是：AI 會不會**編造**「我查過第 142 行的 `ISNULL`」——但那行根本沒這段邏輯？LLM 確實很會生這種看起來很可信的假證據。所以這套框架的底線**不靠 AI 的良心，靠機器**：

- **會執行的 SQL 斷言（硬，造不了假）** — 核心的驗證規則是**真的會在資料庫上跑、回傳違規筆數**的 SQL（見 [9 階段 Phase C → V0.5](9-phase-workflow.md)）。結果是人／腳本跑出來的，AI 沒辦法用嘴帶過。
- **引用查核（硬）** — 對照表裡 AI 引用的行號和程式碼片段，有一支**不靠 AI 的機器檢查**去驗「這行真的存在 + 片段和實際程式碼 token 重疊 ≥ 0.3」——剛好擋掉「虛構第 142 行」這種假證據。
- **15+1 維度自證表（軟）** — 維度層那些文字自證，目前是靠 reviewer 加人工抽查，**還沒有獨立的 agent 去自動 fact-check**（見下面的限制）。

講白一點：**資料層和引用層是機器硬驗、造不了假；敘述層還是有 AI 信任的成分在。** 這條界線我們不含糊。

## 已知限制 | Known Limits

這套框架不是萬靈丹。以下是已知、還沒完全解決的弱點：

1. **注意力 vs 輸入體積** — 在很大的輸入上一次要顧很多維度，LLM 會有「中段注意力耗損（Lost in the Middle）」的風險。目前的緩解是：audit 只看改寫過的區段、維度照撞點挑、太大的段落就分段——**但還沒有 benchmark 能證明它在好幾千行上同時精準咬住全部維度**，所以這點我們不宣稱。
2. **audit 敘述還沒全自動 fact-check** — 維度自證表的文字目前靠人工抽查，還沒有第三個 agent 去 parse、反查每一條自證是不是真的。
3. **斷言規則是 LLM 寫的** — SQL 斷言會被執行（結果是硬的），但規則本身是 LLM 寫出來的，有可能寫得很淺、甚至恆真；規則品質目前靠人工加 verifier 模式覆核，**還沒用突變測試去自動驗證規則到底有沒有效**。

> 會把限制攤出來寫，本來就是這套框架的精神：「**看起來對 ≠ 真的對**」——這句話也適用在框架對自己身上。
